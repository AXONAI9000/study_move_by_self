### 🔑 核心知识点

#### 1. Aptos Token Standard v2
- **对象模型**: 基于 Object 而非账户资源
- **可组合性**: 更灵活的 NFT 组合和扩展
- **引用系统**: 通过对象引用管理 NFT
- **可编程性**: 支持动态属性和行为

#### 2. Collection 核心概念
- **Collection 对象**: NFT 的容器和分组单位
- **创建者权限**: 谁可以铸造和管理
- **版税系统**: 自动版税分配
- **最大供应量**: 限量发行控制

#### 3. Token 对象结构
- **TokenData**: 不可变的元数据（名称、描述、URI）
- **Token**: 实际的 NFT 实例
- **Property Map**: 可变属性系统
- **Object 引用**: 唯一标识和所有权

#### 4. 元数据管理
- **URI 标准**: 指向链下元数据
- **Property Map**: 链上动态属性
- **描述信息**: 名称、描述、图片
- **扩展属性**: 自定义属性系统

5. **案例研究**: 分析现有 NFT 项目

### 常见误区

❌ **误区 1**: 认为 Token v2 只是 v1 的简单升级  
✅ **正确**: v2 是完全重新设计，基于对象模型

❌ **误区 2**: Collection 只是分组  
✅ **正确**: Collection 是独立对象，有自己的生命周期和权限

❌ **误区 3**: NFT 元数据必须全部上链  
✅ **正确**: 通常只存储 URI，实际数据在链下（IPFS 等）

❌ **误区 4**: 所有 NFT 都是不可变的  
✅ **正确**: 通过 Property Map 可以实现可变属性

## 🔥 实战技巧

### Collection 设计最佳实践

```
1. 命名规范
   - Collection Name: 清晰、独特
   - Symbol: 简短、易记（如 "BAYC"）
   
2. 版税设置
   - 通常 2.5% - 10%
   - 考虑二级市场流动性
   
3. 供应量控制
   - 限量 vs 无限量
   - 预留 vs 公开铸造
   
4. 元数据管理
   - IPFS 存储
   - 可变 vs 不可变
```

### NFT 元数据结构

```json
{
  "name": "Cool NFT #1",
  "description": "A really cool NFT",
  "image": "ipfs://QmXxx...",
  "attributes": [
    {
      "trait_type": "Background",
      "value": "Blue"
    },
    {
      "trait_type": "Rarity",
      "value": "Legendary"
    }
  ]
}
```
